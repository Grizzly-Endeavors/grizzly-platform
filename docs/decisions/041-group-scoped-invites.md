# 041: Group-scoped invites and the membership taxonomy

**Date:** 2026-06-29
**Status:** accepted
**Relates to:** [ADR-040](040-invite-broker-cookie-bridged-enrollment.md), [ADR-039](039-authentik-social-federation-invitation-enrollment.md), [ADR-038](038-nextcloud-on-foundation-stores-and-sso.md), [ADR-037](037-authentik-config-as-code-blueprints.md), [ADR-033](033-central-identity-authentik.md)

## Context

ADR-040 made enrollment a one-link flow: an admin mints an invite, the invitee opens `invite.bearflinn.com/i/<token>`, signs in with any social provider, and the enrollment gate consumes the invite via `/verify`. But every enrollee lands in exactly one group — `grizzly-users`, hardcoded as the user-write stage's static `create_users_group`. There was no group taxonomy and no way for an invite to place a person into a specific circle.

The platform serves friends plus two distinct households at varying levels of power. That needs (a) groups beyond the two baseline RBAC ones, and (b) a way to drop a new user into the right group(s) at sign-up without hand-editing them in the Authentik UI afterwards. Manually assigning membership per user is the recurring toil this ADR removes.

A secondary nit: the invite link token was two concatenated UUIDs (64 hex chars), making a ~94-char URL that is awkward to share.

## Decision

### 1. A small, static membership taxonomy in git

Four groups join the two baseline ones in `kubernetes/infrastructure/authentik/blueprints/groups.yaml`: `Big Bear Hollow`, `Fliguntons` (households) and `Rainy Tree Society`, `Nerds` (friend circles). Group creation is rare and structural, so it stays declarative config (stateless upsert blueprints), not a runtime "mint a group" system.

**Membership is orthogonal to power.** A person is `Big Bear Hollow` *plus* a future `household-admin`, never an exploded `big-bear-hollow-admin`. Authentik group memberships stack, so apps map a membership group to access and a (later) power group to elevated capability independently. Power/admin-tier groups are deferred until needed.

**Names are verbatim, with spaces.** An Authentik group's `name` is both identifier and label and propagates unchanged into the OIDC `groups` claim and thus the Nextcloud group name. The only friction — quoting a spaced name in `occ`/CLI — is trivial, and human-readable labels are worth it downstream.

### 2. The invite carries the group(s); the ledger holds them, not the cookie

The `grizzly-invite` broker gains a `groups TEXT[]` column (migration `0002`). `POST /api/invites` accepts an optional `groups[]`; `/verify` returns the group names on a successful consume. The group scope lives in the broker's ledger and travels back only on `/verify` — deliberately **not** in the cookie, which stays a minimal, PII-free signed `{invite_id, exp}`. This preserves ADR-040's invariant (the OAuth-surviving cookie carries no identity and no provider binding) while still letting the scope reach Authentik server-to-server.

### 3. The enrollment gate injects the groups into the flow plan

The `grizzly-invite-gate` expression policy already writes `request.context["prompt_data"]` (for the derived username), which persists into the flow plan. It now also reads the `groups` from the `/verify` response, resolves them to Authentik `Group` objects, and sets `request.context["groups"]`. The user-write stage adds any groups under the plan's `groups` context key on top of its static `create_users_group`. So an enrollee ends up in `grizzly-users` **plus** the invite's groups. Unknown names filter to empty — fail-soft, the user simply isn't added.

### 4. Shorter link token

The link token drops from two concatenated UUIDs (64 hex chars) to a ~12-char base62 string (~71 bits of entropy), generated by rejection-sampling CSPRNG bytes. That is ample for a single-use, status-gated, short-TTL invite, and roughly halves the shared URL. The `token` column stays `UNIQUE` as the collision backstop.

## Alternatives Considered

- **Group membership over OIDC into Nextcloud subadmin.** Wanted: `household-admin` automatically granting group-admin over the household's Nextcloud group. Rejected as out of scope: `user_oidc` mirrors the `groups` claim into Nextcloud *membership* only; subadmin is a separate Nextcloud-side relation no standard OIDC path populates. Per-household subadmin is assigned by hand (rare, set-once) — documented convention, not automation.
- **Carry the groups in the cookie.** Rejected: it would put authorization data in the browser-held token and grow the cookie for no benefit; the ledger is already the single source of truth and `/verify` already round-trips to it.
- **A runtime group-management system.** Rejected: groups change a couple times a year; a 2-line blueprint PR is the right cost. Building minting UX for that is over-engineering.

## Consequences

- **Onboarding a pre-scoped person is one API call.** `POST /api/invites` with `{"groups": ["Rainy Tree Society"]}` → send the short link. No Authentik UI step, no follow-up session.
- **Backward compatible.** An un-scoped invite still enrolls into `grizzly-users` only; pre-existing invite rows default to an empty `groups` array.
- **Live-validation dependency (same posture as ADR-040).** Group injection rides the same `request.context` → flow-plan propagation already proven for the username. It still gets a real end-to-end check (sign up a test identity, confirm dual membership) before being relied on.
- **Cross-repo, explicit split.** `grizzly-invite` owns the ledger column + API; `grizzly-platform` owns the group blueprints and the gate-policy wiring — the same split ADR-040 established. Deploy the broker first so a scoped mint works the moment the policy is live.
- **Nextcloud (and future apps) decide what a group name means.** Nextcloud mirrors names to membership; power semantics stay app-specific and mostly manual. A group name is a signal to key provisioning off, not a uniform grant of rights everywhere.
