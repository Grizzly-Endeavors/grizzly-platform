# Architectural Decision Records

The permanent record of *why* the platform is the way it is. When a non-obvious choice is made, it gets an ADR here (use the `/adr` skill). Start here when you're about to change something and need to know why it was built that way — the rationale, the alternatives that were weighed, and the constraints in play are what an ADR preserves that the code cannot.

See [`INDEX.md`](INDEX.md) for the full list grouped by domain. Files are numbered in rough creation order; a few early numbers repeat (two 003s, two 004s) from before numbering settled — the slug disambiguates. A superseded decision keeps its file and is marked superseded in-place rather than deleted, so the history stays legible.
