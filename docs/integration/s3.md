# Integration: S3 object storage (versitygw)

**What you get:** a scoped S3 account that owns one or more buckets on a foundation gateway, addressed with any S3 SDK:

```
s3-hot:  http://10.0.0.200:7070   (ZFS-backed, latency-sensitive)
s3-bulk: http://10.0.0.200:7072   (MergerFS+SnapRAID, capacity / write-once)
```

Both are [versitygw](https://github.com/versity/versitygw) gateways — a stateless S3-to-POSIX translation layer; each object is exactly one file on disk (ADR-055). Path-style addressing, region `us-east-1`.

## Which gateway?

| | **s3-hot** (`:7070`) | **s3-bulk** (`:7072`) |
|---|---|---|
| Backing | ZFS pool | MergerFS + SnapRAID |
| Best for | Small, frequently-read, latency-sensitive objects; app working sets | Large, append/write-once objects; archives, snapshots, transcripts, backups |
| Cost profile | Fast, smaller | Cheap, big, parity-protected |

When in doubt for bulk app data (page snapshots, LLM transcripts, media), use **s3-bulk** — that's what career-scanner and the registry use. Reserve **s3-hot** for genuinely latency-sensitive working sets.

## When to use it

- **Use it** for blobs, files, uploads, backups, or anything you'd reach for object storage for — never store large binaries in Postgres or Valkey.
- Need durable relational state? [Postgres](postgres.md). Ephemeral cache? [Valkey](valkey.md).

## Prerequisites

- The target gateway running (`deploy-foundation-stores.yml`), and its **root** creds seeded in OpenBao (`stores/s3-hot` or `stores/s3-bulk`, keys `root_access_key`/`root_secret_key`).
- A per-app access/secret pair seeded at `secret/grizzly-platform/stores/<app>` (keys `s3_access_key`, `s3_secret_key`) — you generate these and write them before provisioning.

## 1 — Provision the account + bucket

Provisioning uses the shared `tasks/versitygw-provision-account.yml`, called from your app's `setup-<app>-stores.yml` (the S3 block of `setup-career-scanner-stores.yml` is the copy-me reference). It creates a **`userplus`** account (a role that may create and own its own buckets — plain `user` cannot) and its bucket on the chosen gateway:

```yaml
- name: Provision the <app> account + bucket on s3-bulk
  ansible.builtin.include_tasks: "{{ playbook_dir }}/../tasks/versitygw-provision-account.yml"
  vars:
    vgw_instance: s3-bulk
    vgw_root_access: "{{ vault_s3_bulk_root_access_key }}"
    vgw_root_secret: "{{ vault_s3_bulk_root_secret_key }}"
    vgw_access: "{{ vault_<app>_s3_access_key }}"
    vgw_secret: "{{ vault_<app>_s3_secret_key }}"
    vgw_bucket: <app>
```

```bash
ansible-playbook -i ansible/inventory ansible/playbooks/setup-<app>-stores.yml \
  --vault-password-file .vault_pass --tags s3 -v
```

Accounts live in OpenBao (versitygw's Vault-IAM backend), not on the gateway — so they survive gateway restarts. **Note the IAM cache:** account *changes* take up to 120s to take effect (`--iam-cache-ttl`); a brand-new key works immediately on first use.

## 2 — Wire it into your app

Land both keys with an `ExternalSecret` ([secrets.md](secrets.md)):

```yaml
data:
  - secretKey: S3_ACCESS_KEY
    remoteRef: { key: grizzly-platform/stores/<app>, property: s3_access_key }
  - secretKey: S3_SECRET_KEY
    remoteRef: { key: grizzly-platform/stores/<app>, property: s3_secret_key }
```

Then configure your SDK — **endpoint override + path-style addressing** are the two things every SDK needs against a non-AWS gateway:

```python
# boto3
s3 = boto3.client(
    "s3",
    endpoint_url="http://10.0.0.200:7072",     # s3-bulk
    aws_access_key_id=os.environ["S3_ACCESS_KEY"],
    aws_secret_access_key=os.environ["S3_SECRET_KEY"],
    region_name="us-east-1",
    config=Config(s3={"addressing_style": "path"}),
)
```

```bash
# aws-cli — path-style is automatic with --endpoint-url
AWS_ACCESS_KEY_ID=$S3_ACCESS_KEY AWS_SECRET_ACCESS_KEY=$S3_SECRET_KEY \
  aws --endpoint-url http://10.0.0.200:7072 s3 ls s3://<app>/
```

## Verify

```bash
aws --endpoint-url http://10.0.0.200:7072 s3 cp ./x.txt s3://<app>/health/x.txt
aws --endpoint-url http://10.0.0.200:7072 s3 ls s3://<app>/health/
```

## Troubleshoot

- **`SignatureDoesNotMatch`** — wrong secret key, or the SDK is using virtual-host addressing. Force `addressing_style: path` (SDK) / use `--endpoint-url` (aws-cli). Bucket-name validation runs *before* signature check, so an invalid bucket name can masquerade as an auth error — test against a valid name.
- **`AccessDenied` on `CreateBucket`** — the account is a plain `user`, not `userplus`. Re-provision with `userplus` (the shared task does this).
- **New/changed key not working for ~2 min** — IAM cache (`--iam-cache-ttl`, 120s) on account *changes*. Wait it out.
- **Can't reach the port from a pod** — gateways bind the host LAN interface; reach them at `10.0.0.200:7070/7072` over the flat L2, not via a K8s Service.
